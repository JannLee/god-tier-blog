# Next.js + GitHub Pages로 기술 블로그 만들기

안녕하세요! 이 블로그는 **Next.js 16 App Router**와 **GitHub Pages**를 사용해 만들었습니다. 단순히 글을 쓰는 공간을 넘어, 직접 설계하고 배포까지 경험하는 사이드 프로젝트로 시작했습니다.

## 왜 블로그를 만들었나요?

개발하면서 배운 것들을 정리하고 공유하고 싶었습니다. 노션이나 티스토리 같은 플랫폼도 좋지만, 직접 만든 블로그는 자유도가 다릅니다. 코드 하이라이팅, 다크 모드, SEO 구성 등 모든 것을 내 손으로 결정할 수 있습니다.

## 기술 스택

- **프레임워크**: Next.js 16 (App Router)
- **스타일링**: Tailwind CSS v4
- **콘텐츠**: MDX
- **배포**: GitHub Pages (정적 내보내기)

### Next.js를 선택한 이유

Next.js 16의 App Router는 정적 내보내기(`output: 'export'`)와 완벽하게 결합됩니다. 빌드 결과물은 순수 HTML/CSS/JS 파일이라 서버가 전혀 필요 없고, CDN에서 바로 서빙할 수 있습니다. Lighthouse 성능 점수 90+ 달성이 목표였는데, 정적 사이트 특성상 서버 응답 지연이 없어 Core Web Vitals 최적화가 수월합니다.

### Tailwind CSS v4를 선택한 이유

Tailwind CSS v4는 기존 v3 대비 빌드 속도가 **3.5배 향상**되었습니다(공식 벤치마크 기준). 또한 CSS-first 설정 방식으로 `tailwind.config.js` 파일 없이 CSS 변수만으로 테마를 정의할 수 있어, 설정 파일 관리 부담이 줄어들었습니다.

### MDX를 선택한 이유

MDX는 마크다운과 JSX를 혼용할 수 있는 포맷입니다. 일반 마크다운으로 글을 쓰면서도 필요할 때 React 컴포넌트를 삽입할 수 있습니다. Shiki 기반 코드 하이라이팅, 커스텀 콜아웃 박스, 인터랙티브 예제 등을 콘텐츠 안에 직접 넣을 수 있는 게 큰 장점입니다.

### GitHub Pages를 선택한 이유

월 비용이 **$0**입니다. 커스텀 도메인도 지원하고, GitHub Actions와 연동하면 `main` 브랜치에 push할 때마다 자동으로 빌드·배포됩니다. 별도 서버나 호스팅 서비스 없이 완전한 CI/CD 파이프라인을 구축할 수 있습니다.

## 실제 성능 측정값

정적 사이트의 가장 큰 장점은 성능입니다. 현재 달성 목표 및 측정값:

| 지표 | 목표값 | 달성값 |

## 요약

- Next.js 16 App Router + 정적 내보내기로 서버 없이 운영 가능
- Tailwind CSS v4로 빌드 속도 3.5배 향상, 설정 파일 불필요
- MDX로 마크다운 + React 컴포넌트 혼용 가능
- GitHub Pages + GitHub Actions로 배포 비용 $0, 자동 CI/CD 구축

## FAQ

### Q. 왜 Velog나 티스토리를 안 썼나요?

Velog와 티스토리는 훌륭한 플랫폼이지만, 디자인·기능 커스터마이징에 한계가 있습니다. 직접 만든 블로그는 코드 하이라이팅 테마, 사이드바 구조, SEO 메타 태그 등 모든 것을 원하는 대로 제어할 수 있습니다. 또한 블로그를 만드는 과정 자체가 좋은 학습 경험이었습니다.

### Q. 블로그 만드는 데 얼마나 걸렸나요?

기본적인 글 목록, 포스트 렌더링, 배포 파이프라인까지 약 2주 정도 걸렸습니다. 다크 모드, ToC(목차), 코드 복사 버튼, 검색 기능 등 부가 기능들은 그 이후에 점진적으로 추가했습니다.

### Q. 코드 블록 하이라이팅은 어떻게 구현했나요?

[Shiki](https://shiki.matsu.io/)를 사용했습니다. `next-mdx-remote` 또는 `@next/mdx`와 함께 `rehype-shiki` 플러그인을 연결하면 빌드 타임에 서버에서 하이라이팅 처리가 완료됩니다. 클라이언트 사이드 JS 없이 완벽한 구문 강조를 제공합니다.

### Q. 다크 모드는 어떻게 구현했나요?

Tailwind CSS의 `dark:` 변형 클래스와 `next-themes` 라이브러리를 조합했습니다. `localStorage`에 테마 설정을 저장하고, `ThemeProvider`로 전체 앱을 감싸 SSR 하이드레이션 불일치 없이 동작하도록 구현했습니다.

### Q. 블로그 소스 코드는 공개되어 있나요?

네, GitHub에 공개되어 있습니다. 레포지토리를 참고하시면 전체 구조와 설정 파일을 확인하실 수 있습니다.
